# 今後の開発ロードマップ (Next Steps)

本プロジェクトの基本実装は完了しました。
ユーザー様より頂いた「データ取得戦略ガイド」に基づき、実運用に向けた詳細なステップを以下に定義します。

## 1. ローカル環境でのデータ収集とモデル構築 (Phase 1)
まずはローカル環境（Mac）で、過去データの蓄積とモデルの精度向上に集中します。VPSはまだ不要です。

- [ ] **静的データの体系的収集 (netkeiba.com)**
    - [ ] **開催日・レースIDのリスト化**: 過去5〜10年分のリストを作成。
    - [ ] **レース結果のスクレイピング**: 数日かけて少しずつ実行し、HTMLをローカルに蓄積する。
    - [ ] **構造化データへの変換**: 蓄積したHTMLをCSVに変換し、学習用データセットを作成する。
    - [ ] **属性データ収集**: 馬・騎手・調教師のデータを追加取得する。

- [ ] **モデルの精度向上 (Model Improvement)**
    - [ ] **特徴量エンジニアリング**: 血統やリーディング順位などを組み込む。
    - [ ] **ハイパーパラメータチューニング**: `Optuna` でLightGBMを最適化。
    - [ ] **厳密なバックテスト**: 過去データを使ったシミュレーションで、収益性を検証する。

## 2. 本番運用環境の構築 (Phase 2)
モデルの有効性が確認でき、実際に自動投票を行いたくなった段階でVPSを導入します。

- [ ] **動的データのリアルタイム取得 (JRA公式サイト)**
    - [ ] Playwrightのスクリプトを開発し、ローカルで動作確認する。
    - [ ] **注**: この段階ではまだVPSは必須ではありませんが、本番を見据えて実装します。

- [ ] **自動化とインフラ構築 (VPS & Scheduler)**
    - [ ] **VPS環境のセットアップ**: ConoHa VPS等の契約とセキュリティ設定。
    - [ ] **スケジューリング**: 24時間稼働の設定。
    - [ ] **自動投票の開始**: 少額からの運用スタート。

## 3. 投票戦略の洗練 (Strategy Refinement)
運用しながら継続的に改善します。

- [ ] **資金管理 (Money Management)**
    - [ ] 定額購入から、期待値に応じた可変金額（ケリー基準など）への移行。
    - [ ] 1日の最大損失額を設定するサーキットブレーカーの実装。
